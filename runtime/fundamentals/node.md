Deno runs Node.js code: built-in Node modules, npm packages, `package.json`,
CommonJS, and native addons are all supported. This page takes one small script
and grows it until it has touched all of those pieces. Here is the starting
point, a little health report that uses an npm package (`chalk`), a Node
built-in module (`node:os`), and a Node global (`process`):

```js title="report.mjs"
import os from "node:os";
import chalk from "npm:chalk@5";

console.log(chalk.green(`${os.hostname()} is up`));
console.log(`load average: ${os.loadavg().map((n) => n.toFixed(2)).join(" ")}`);
console.log(`free memory: ${(os.freemem() / 2 ** 30).toFixed(1)} GiB`);
console.log(`report generated by pid ${process.pid}`);
```

```sh
$ deno run --allow-env --allow-sys report.mjs
MacBookPro.localdomain is up
load average: 6.28 8.58 6.32
free memory: 2.3 GiB
report generated by pid 82304
```

Three Node-specific things worked. Deno fetched `chalk` from npm on the first
run and cached it, the `node:os` import went through Deno's Node compatibility
layer, and `process` was available as a global. The permission flags are the one
visibly Deno thing here: npm packages and Node built-ins are subject to the same
[permission system](/runtime/fundamentals/security/) as the rest of your code
(`chalk` reads environment variables, `node:os` reports system information).

The compatibility layer is broad: as of Deno 2.8, **over 75% of Node's own test
suite passes** in Deno, covering nearly every `node:` module. You can track the
current state at [node-test-viewer.deno.dev](https://node-test-viewer.deno.dev/)
and browse the [list of supported Node.js APIs](/runtime/reference/node_apis/).

## Using Node's built-in modules

Suppose the first import in `report.mjs` had been written without the prefix, as
most existing Node code is:

```js title="report.mjs"
import os from "os";
```

```sh
$ deno run report.mjs
error: Import "os" not a dependency
  hint: If you want to use a built-in Node module, add a "node:" prefix (ex. "node:os").
    at file:///tmp/report/report.mjs:1:16
```

Deno requires the `node:` specifier for Node built-ins. The same hints, with
quick-fixes, are provided by the Deno LSP in your editor. Once the prefix is in
place, code using Node built-ins behaves as it does in Node.js, and the `node:`
form also works in Node itself, so updated files stay portable.

The `node:module` built-in includes the
[`registerHooks()`](/runtime/reference/loader_hooks/) API for customizing module
resolution and loading from inside your program.

<a href="/api/node/" class="docs-cta runtime-cta">Explore built-in Node APIs</a>

## Using npm packages

The report's other import, `npm:chalk@5`, is an npm specifier. These import npm
packages directly, with this format:

```console
npm:<package-name>[@<version-requirement>][/<sub-path>]
```

For a single-file script like the report, no install command runs beforehand and
no `node_modules` folder is created: the dependency is downloaded into Deno's
global cache on first use. Later sections cover the cases where that is not
enough (a `package.json` project, native addons), each with a one-step fix.

npm specifiers also cover what `npx` does in a Node workflow: where you would
run `npx create-next-app@latest`, run a package straight from the registry with:

```sh
deno run -A npm:create-next-app@latest
```

For examples with popular libraries, refer to the
[tutorial section](/runtime/tutorials).

### Run npm binaries

A package's `bin` entries can be run directly from an npm specifier, with an
optional binary name as the sub-path:

```sh
$ deno run -R npm:cowsay@1.5.0 "Hello there!"
 ______________
< Hello there! >
 --------------
        \   ^__^
         \  (oo)\_______
            (__)\       )\/\
                ||----w |
                ||     ||
```

Select a different binary from the same package with the sub-path:
`deno run -R npm:cowsay@1.5.0/cowthink "What to eat?"`.

## Node.js global objects

The report script used `process.pid` without importing anything. Node.js defines
a number of [global objects](https://nodejs.org/api/globals.html), and Deno
provides the common ones:

- `process` is available to all code; it is by far the most used Node global in
  npm packages. If you prefer explicit imports, enable the
  [`no-process-global`](/lint/rules/no-process-global/) lint rule (off by
  default since Deno 2.8) and `deno lint` will flag uses of the global:

  ```js title="process.js"
  console.log(process.versions.deno);
  ```

  ```shell
  $ deno lint process.js
  error[no-process-global]: NodeJS process global is discouraged in Deno
   --> /process.js:1:13
    |
  1 | console.log(process.versions.deno);
    |             ^^^^^^^
    = hint: Add `import process from "node:process";`

  Found 1 problem (1 fixable via --fix)
  Checked 1 file
  ```

- `require()` is supported in CommonJS files; see
  [CommonJS support](#commonjs-support).

- `Buffer` is not a global. Import it explicitly from `node:buffer`:

  ```js title="buffer.js"
  import { Buffer } from "node:buffer";

  const buf = new Buffer(5, "0");
  ```

  Prefer
  [`Uint8Array`](https://developer.mozilla.org/en-US/docs/Web/JavaScript/Reference/Global_Objects/Uint8Array)
  or other
  [`TypedArray`](https://developer.mozilla.org/en-US/docs/Web/JavaScript/Reference/Global_Objects/TypedArray)
  subclasses for new code.

- `__filename` and `__dirname` have ESM replacements: use `import.meta.filename`
  and `import.meta.dirname`.

- `setTimeout` / `setInterval`: starting in Deno 2.8, the global timer functions
  return a Node.js [`Timeout`](https://nodejs.org/api/timers.html#class-timeout)
  object instead of a number, matching Node.js semantics. The returned object
  exposes methods like `.ref()`, `.unref()`, `.refresh()`, and `.hasRef()`, and
  still coerces to a number (via `Symbol.toPrimitive`), so existing code that
  stores the timer ID as a number or passes it to `clearTimeout`/`clearInterval`
  works unchanged.

  ```ts
  const t = setTimeout(() => {}, 1000);
  t.unref(); // don't keep the event loop alive for this timer
  clearTimeout(t);
  ```

## First-class package.json support

Node projects declare dependencies in a `package.json` rather than pinning
versions inline. Let's move the report there, with a script entry:

```json title="package.json"
{
  "name": "report",
  "type": "module",
  "scripts": {
    "report": "deno run --allow-env --allow-sys report.mjs"
  },
  "dependencies": {
    "chalk": "^5.6.0"
  }
}
```

```js title="report.mjs"
import os from "node:os";
import chalk from "chalk";
// ...
```

Run it again and you hit the first real edge:

```sh
$ deno run --allow-env --allow-sys report.mjs
error: Could not resolve "chalk", but found it in a package.json. Deno expects the node_modules/ directory to be up to date. Did you forget to run `deno install`?
    at file:///tmp/report/report.mjs:2:19
```

A bare specifier like `"chalk"` resolves through `package.json`, and a project
with a `package.json` defaults to the `manual` node_modules mode: you run an
install step, and dependencies live in a local `node_modules` directory. The
error tells you the fix:

```sh
$ deno install
Initialize chalk@5.6.2
Installed 1 package in 2ms

Dependencies:
+ npm:chalk 5.6.2
```

This creates `node_modules` and a `deno.lock` lockfile. (To skip the install
step, set `"nodeModulesDir": "auto"`; the [node_modules](#node_modules) section
compares the modes.) Now the script runs, and so does the `package.json` script,
via `deno task`:

```sh
$ deno task report
Task report deno run --allow-env --allow-sys report.mjs
MacBookPro.localdomain is up
...
```

Deno also reads `package.json` fields like `"type"` when deciding how to load
files, which matters in the next step.

## CommonJS support

Say the report's uptime formatting already exists in an old CommonJS helper,
copied in from a legacy codebase:

```js title="legacy/format-uptime.js"
const os = require("node:os");

module.exports = function formatUptime() {
  const minutes = Math.floor(os.uptime() / 60);
  return `${Math.floor(minutes / 60)}h ${minutes % 60}m`;
};
```

```sh
$ deno run legacy/format-uptime.js
error: Uncaught (in promise) ReferenceError: require is not defined
const os = require("node:os");
           ^
    at file:///tmp/report/legacy/format-uptime.js:1:12

    info: Deno supports CommonJS modules in .cjs files, or when the closest
          package.json has a "type": "commonjs" option.
    hint: Rewrite this module to ESM,
          or change the file extension to .cjs,
          or add package.json next to the file with "type": "commonjs" option,
          or pass --unstable-detect-cjs flag to detect CommonJS when loading.
    docs: https://docs.deno.com/go/commonjs
```

Because of the `.js` extension and the `"type": "module"` in our `package.json`,
Deno loaded the file as an ES module, where `require` and `module` do not exist.
The error lists every fix; the least invasive one here is the rename:

```sh
mv legacy/format-uptime.js legacy/format-uptime.cjs
```

CommonJS modules can be imported from ES modules directly, so the report can use
the helper as-is:

```js title="report.mjs"
import os from "node:os";
import chalk from "chalk";
import formatUptime from "./legacy/format-uptime.cjs";

console.log(chalk.green(`${os.hostname()} is up ${formatUptime()}`));
```

```sh
$ deno run --allow-env --allow-sys report.mjs
MacBookPro.localdomain is up 6h 46m
```

### How Deno determines module type

What the error message above hinted at, spelled out:

- A `.cjs` file is always CommonJS; `package.json` is not consulted.
- `.js`, `.jsx`, `.ts`, and `.tsx` files are loaded as CommonJS when the closest
  `package.json` (next to the file or up the directory tree) contains
  `"type": "commonjs"`. Tools like Next.js's bundler generate such a
  `package.json` automatically, and adding the option to an existing CommonJS
  project makes it work in both Node.js and Deno.
- Deno does not otherwise analyze module contents to detect CommonJS, because
  scanning the file system for `package.json` files and analyzing modules takes
  longer than not doing it, and to discourage the use of CommonJS. You can opt
  into detection with the `--unstable-detect-cjs` flag in Deno >= 2.1.2. It
  takes effect except when there's a `package.json` file with
  `{ "type": "module" }`.

### Using require()

CommonJS code can `require()` npm packages, with two practical requirements:
dependencies must be installed into a local `node_modules` directory (`auto` or
`manual` mode), and the code typically needs the `-R` (`--allow-read`) and `-E`
(`--allow-env`) flags, because Deno probes `package.json` files and the
`node_modules` directory to resolve CommonJS modules:

```js title="main.cjs"
const express = require("express");
```

```shell
$ deno install npm:express
Dependencies:
+ npm:express 5.2.1

$ deno run -R -E main.cjs
[Function: createApplication] {
  application: { init: [Function: init], ... }
}
```

In ES modules, you can create an instance of the `require()` function manually,
with the same install and permission requirements:

```js title="main.js"
import { createRequire } from "node:module";
const require = createRequire(import.meta.url);
const express = require("express");
```

### CommonJS and ESM interop

Deno's `require()` implementation supports requiring ES modules. This works the
same as in Node.js, where you can only `require()` ES modules that don't have
top-level await in their module graph.

```js title="greet.js"
export function greet(name) {
  return `Hello ${name}`;
}
```

```js title="esm.js"
import { greet } from "./greet.js";

export { greet };
```

```js title="main.cjs"
const esm = require("./esm");
console.log(esm);
console.log(esm.greet("Deno"));
```

```shell
$ deno run -R main.cjs
[Module: null prototype] { greet: [Function: greet] }
Hello Deno
```

The other direction, importing CommonJS from ES modules, is what the report
script already does with its `.cjs` helper: the default import receives whatever
the module assigned to `module.exports`.

### Troubleshooting

Deno will guide you when a file looks like CommonJS but isn't loaded as such.
Whenever you see an error about `module` or `require` not being defined, the
four fixes from the error message earlier in this section apply: rewrite the
code to ESM, change the file extension to `.cjs`, add a nearby `package.json`
with `{ "type": "commonjs" }`, or run with `--unstable-detect-cjs`.

## node_modules

Our project picked up a `node_modules` directory the moment it gained a
`package.json`. Whether you need one at all depends on which mode you are in:

| Mode   | When to use                                  | How to enable                                              |
| ------ | -------------------------------------------- | ---------------------------------------------------------- |
| none   | Most Deno projects; keep repo clean          | Default; do nothing                                        |
| auto   | Tools/bundlers expect node_modules; Node-API | `"nodeModulesDir": "auto"` or `--node-modules-dir=auto`    |
| manual | Existing package.json with install step      | `"nodeModulesDir": "manual"` + run `deno install`/npm/pnpm |

:::note

We recommend that you use the default `none` mode, and fall back to `auto` or
`manual` mode if you get errors about missing packages inside the `node_modules`
directory.

:::

In the default `none` mode, npm packages resolve to a central global cache, the
way the single-file examples at the top of this page worked. This uses less
space and keeps your project directory clean. The other two modes exist because
parts of the npm ecosystem (bundlers, frameworks, packages with native addons)
read `node_modules` from disk and break without it.

### Automatic node_modules creation

`auto` mode installs dependencies into the global cache and creates a local
`node_modules` directory in the project root on each run, with no separate
install step. It is recommended for projects with npm dependencies that rely on
the `node_modules` directory: mostly projects using bundlers, or ones with npm
dependencies that have postinstall scripts. Enable it per command with
`deno run --node-modules-dir=auto main.ts`, or in config:

```json title="deno.json"
{
  "nodeModulesDir": "auto"
}
```

### Manual node_modules creation

`manual` mode is what our report project has been using since it gained a
`package.json`: it is the default in that case, and it requires an explicit
install step (`deno install`, `npm install`, or any other package manager). You
may recognize this workflow from Node.js projects. It is recommended for
projects using frameworks like Next.js, Remix, Svelte, or Qwik, and tools like
Vite, Parcel, or Rollup. Enable it per command with
`deno run --node-modules-dir=manual main.ts`, or in config:

```json title="deno.json"
{ "nodeModulesDir": "manual" }
```

### node_modules layout: isolated vs hoisted

When a local `node_modules` directory exists, Deno can lay it out in two ways.
The default (**isolated**) installs each package into a content-addressed
`.deno/` directory and exposes it through a symlink, so every package only sees
its declared dependencies. This is similar to pnpm's layout.

```text
node_modules/
├── .deno/chalk@5.6.2/node_modules/chalk/   ← real files
└── chalk -> .deno/chalk@5.6.2/node_modules/chalk
```

Some npm tooling, and any package that walks `node_modules` looking for
flat-resolved siblings, assumes the **hoisted** layout that npm and Yarn classic
use. Deno 2.8 adds a hoisted mode
([denoland/deno#32788](https://github.com/denoland/deno/pull/32788)). The
hoisted linker requires a manually-managed `node_modules` directory, so set
`nodeModulesDir` to `manual`:

```json title="deno.json"
{
  "nodeModulesDir": "manual",
  "nodeModulesLinker": "hoisted"
}
```

Or as a one-off CLI flag:
`deno install --node-modules-dir=manual --node-modules-linker=hoisted`. In
hoisted mode the most-depended-upon version of each package is placed at the top
of `node_modules/`, and conflicting versions are nested under the dependent that
needs them, just like npm.

Stick with the default isolated mode unless a tool you depend on requires the
hoisted layout. Isolated mode catches phantom dependencies that hoisted layouts
hide.

## Node-API addons

Time to give the report a history, stored in DuckDB. `npm:duckdb` is a native
[Node-API addon](https://nodejs.org/api/n-api.html), the last big piece of Node
compatibility. Add it and watch the install:

```sh
$ deno install npm:duckdb
╭ Warning
│
│  Ignored build scripts for packages:
│  npm:duckdb@1.4.4
│
│  Run "deno approve-scripts" to run build scripts.
╰─
```

That warning matters. Many native packages rely on npm lifecycle scripts (for
example, `postinstall`) to download or compile their binding, and Deno does not
run those scripts by default, for security reasons. Ignore the warning and the
addon fails at runtime:

```js title="history.mjs"
import duckdb from "duckdb";

const db = new duckdb.Database(":memory:");
db.all("SELECT 40 + 2 AS answer", (err, rows) => {
  if (err) throw err;
  console.log(rows);
});
```

```sh
$ deno run -R -E --allow-ffi history.mjs
error: Uncaught (in promise) Error: Cannot find module '/tmp/report/node_modules/.deno/duckdb@1.4.4/node_modules/duckdb/lib/binding/duckdb.node'
```

The missing `duckdb.node` file is the native binding that the package's install
script never got to fetch. The fix is to allow that specific package's scripts
(or run the interactive `deno approve-scripts` command):

```sh
$ deno install --allow-scripts=npm:duckdb
Initialize duckdb@1.4.4: running 'install' script

$ deno run -R -E --allow-ffi history.mjs
[ { answer: 42 } ]
```

Two requirements to remember: Node-API addons need a local `node_modules`
directory (`auto` or `manual` mode; supported since Deno 2.0), and loading
native code requires the explicit `--allow-ffi` permission, like all
[FFI in Deno](/runtime/reference/permissions/#ffi-(foreign-function-interface)).
The [`deno install` docs](/runtime/reference/cli/install/) cover the
lifecycle-scripts options. Packages that use public Node-APIs, like
[`esbuild`](https://www.npmjs.com/package/esbuild),
[`npm:sqlite3`](https://www.npmjs.com/package/sqlite3), and
[`npm:duckdb`](https://www.npmjs.com/package/duckdb), can be expected to work.

## Importing types

Many npm packages ship with types; `import chalk from "npm:chalk@5"` gives you
typed `chalk` with no extra setup.

Some packages do not ship with types but you can specify their types with the
[`@ts-types`](/runtime/fundamentals/typescript) directive. For example, using a
[`@types`](https://www.typescriptlang.org/docs/handbook/2/type-declarations.html#definitelytyped--types)
package:

```ts
// @ts-types="npm:@types/express@^4.17"
import express from "npm:express@^4.17";
```

### Module resolution

The official TypeScript compiler `tsc` supports different
[moduleResolution](https://www.typescriptlang.org/tsconfig#moduleResolution)
settings; Deno only supports the modern `node16` resolution. Many npm packages
fail to correctly provide types under node16 module resolution, which can result
in `deno check` reporting type errors that `tsc` does not report.

If a default export from an `npm:` import appears to have a wrong type (with the
right type seemingly available under the `.default` property), the package most
likely provides wrong types under node16 module resolution for imports from ESM.
You can verify this by checking if the error also occurs with
`tsc --module node16` and `"type": "module"` in `package.json`, or by consulting
the [Are the types wrong?](https://arethetypeswrong.github.io/) website
(particularly the "node16 from ESM" row).

If you want to use a package that doesn't support TypeScript's node16 module
resolution, you can:

1. Open an issue at the issue tracker of the package about the problem, and
   perhaps contribute a fix. (Unfortunately, there is a lack of tooling for
   packages to support both ESM and CJS, since default exports require different
   syntaxes. See also
   [microsoft/TypeScript#54593](https://github.com/microsoft/TypeScript/issues/54593))
2. Use a [CDN](/runtime/fundamentals/modules/#url_imports) that rebuilds the
   packages for Deno support, instead of an `npm:` identifier.
3. Ignore the type errors you get in your code base with `// @ts-expect-error`
   or `// @ts-ignore`.

## Including Node types

Starting in Deno 2.8, `deno check` and the LSP include `lib.node` in every
type-check by default, so Node ambient types like `Buffer`, `NodeJS.Timeout`,
and `process` resolve without any configuration:

```ts
// 2.8+: type-checks with no extra setup
const buf: Buffer = Buffer.from("hello");
const t: NodeJS.Timeout = setTimeout(() => {}, 0);
```

The bundled `lib.node` tracks the major version of `@types/node` that matches
the Node release Deno reports in `process.versions.node`. If you need to pin a
specific `@types/node` version (for example, to match your project's Node
version), add it as an explicit dependency:

```jsonc title="deno.json"
{
  "imports": {
    "@types/node": "npm:@types/node@^22"
  }
}
```

On versions before 2.8, or if you've opted out of `lib.node`, you can still load
the types with a reference directive:

```ts
/// <reference types="npm:@types/node" />
```

Other Node.js-specific types are available through the same `NodeJS` namespace,
for example `NodeJS.BufferEncoding`.

## Conditional exports

Package exports can be
[conditioned](https://nodejs.org/api/packages.html#conditional-exports) on the
resolution mode. The conditions satisfied by an import from a Deno ESM module
are as follows:

```json
["deno", "node", "import", "module-sync", "default"]
```

This means that the first condition listed in a package export whose key equals
any of these strings will be matched. For `require()` resolution, including
`createRequire()`, the conditions are:

```json
["require", "node", "module-sync", "default"]
```

Deno also applies `module-sync` when analyzing CommonJS modules that re-export
through `require()`.

You can expand the import conditions list using the `--conditions` CLI flag. For
example, `deno run --conditions development,react-server main.ts` prepends those
two conditions to the import list above.

## Migrating from Node to Deno

The report project went from a single script to a `package.json`, a CommonJS
helper, and a native addon, and each step needed at most one command or config
line. That is representative: running a Node.js project with Deno usually
requires little to no change. See the
[Migrating from Node.js to Deno guide](/runtime/migrate/) for the details,
optional toolchain improvements, and a Node-to-Deno command cheatsheet.

## Private registries

:::caution

Not to be confused with
[private repositories and modules](/runtime/packages/private_repositories/).

:::

Deno supports private registries, which allow you to host and share your own
modules. This is useful for organizations that want to keep their code private
or for individuals who want to share their code with a select group of people.

### What are private registries?

Large organizations often host their own private npm registries to manage
internal packages securely. These private registries serve as repositories where
organizations can publish and store their proprietary or custom packages. Unlike
public npm registries, private registries are accessible only to authorized
users within the organization.

### How to use private registries with Deno

First, configure your
[`.npmrc`](https://docs.npmjs.com/cli/v10/configuring-npm/npmrc) file to point
to your private registry. The `.npmrc` file must be in the project root or
`$HOME` directory. Add the following to your `.npmrc` file:

```sh
@mycompany:registry=http://mycompany.com:8111/
//mycompany.com:8111/:_authToken=secretToken
```

Replace `http://mycompany.com:8111/` with the actual URL of your private
registry and `secretToken` with your authentication token. `_authToken` is the
standard bearer-token form; registries that use legacy `_auth` credentials are
also supported (see the `.npmrc` features list below).

Then update Your `deno.json` or `package.json` to specify the import path for
your private package. For example:

```json title="deno.json"
{
  "imports": {
    "@mycompany/package": "npm:@mycompany/package@1.0.0"
  }
}
```

or if you're using a `package.json`:

```json title="package.json"
{
  "dependencies": {
    "@mycompany/package": "1.0.0"
  }
}
```

Now you can import your private package in your Deno code:

```typescript title="main.ts"
import { hello } from "@mycompany/package";

console.log(hello());
```

and run it using the `deno run` command:

```sh
deno run main.ts
```

### `.npmrc` configuration

Beyond the basic registry / token setup above, Deno reads several other `.npmrc`
fields. The ones most likely to matter:

- **Mutual-TLS authentication** (Deno 2.8+): `certfile` and `keyfile` point at
  PEM files used to authenticate the client when the registry requires mTLS.

  ```ini title=".npmrc"
  //registry.mycompany.com/:certfile=/etc/deno/client.crt
  //registry.mycompany.com/:keyfile=/etc/deno/client.key
  ```

- **`email` on `_auth` entries** (Deno 2.8+): some legacy on-prem registries
  require an `email` alongside the auth token.

  ```ini title=".npmrc"
  //registry.mycompany.com/:_auth=secretToken
  //registry.mycompany.com/:email=ci@mycompany.com
  ```

- **`min-release-age`** (Deno 2.8+): refuses to install package versions younger
  than the configured age. Useful as a default supply-chain guard for all
  installs. The same control is also available as the CLI flag
  `--minimum-dependency-age` and the `minimumDependencyAge` field in
  `deno.json`. See
  [Minimum dependency age](/runtime/packages/supply_chain/#minimum-dependency-age)
  for the full picture.

  ```ini title=".npmrc"
  min-release-age=3
  ```

- **`NPM_CONFIG_REGISTRY` env var**: overrides the registry set in `.npmrc`,
  matching npm's precedence (handy in CI when you want to redirect installs
  without editing the checked-in `.npmrc`).

### `file:` and `link:` dependencies in published packages

Some published npm packages accidentally ship a `file:` or `link:` specifier in
their `package.json` that points at a path on the publisher's machine:

```jsonc title="some-package/package.json"
{
  "dependencies": {
    "lodash": "^4.17.0",
    "local-helpers": "file:../local-helpers"
  }
}
```

Starting in Deno 2.8, those `file:` and `link:` entries are silently skipped
while resolving npm metadata, so packages that carry a stray local-path
dependency install cleanly instead of failing with an "Invalid version
requirement" error.
